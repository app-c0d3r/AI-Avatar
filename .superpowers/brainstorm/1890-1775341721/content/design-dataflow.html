Design — Section 1: Data Flow

How emotion travels from LLM response to avatar bones

User sends message
──►
POST /api/chat
│
▼
Backend (Python / FastAPI)
1. Appends to system prompt:
  "Start your reply with [EMO:emotion:intensity]
   then your actual response."
2. Streams LLM response
3. Detects [EMO:happy:medium] in first chunk
4. Emits data: {"emotion":"happy","intensity":"medium"}
5. Strips tag, streams remaining content normally
│
▼
ChatInterface.jsx — SSE parser
• If parsed.emotion → store pendingEmotion + pendingIntensity
• Content chunks → accumulate + display as before
• On stream end → dispatchEvent('vrm-pose-change', {pose, intensity})
• Then calls speakText() → triggers talk layer
│
▼
GLTFAvatar.tsx — useFrame (runs at 60fps)
LAYER 1 · Pose
getPoseTargets(pose, intensity, t)
→ lUAz, rUAz, lLAz, rLAz
→ spineX (new)
→ headZ tilt (new)
lerp all at 0.08
LAYER 2 · Talk
Active during TTS audio
sin-wave wrist motion
rLA.x oscillates ±0.1
lUA.z breathes ±0.05
fades out on audio end
LAYER 3 · Existing
Mouse → head/neck
Blink timer
Mouth via analyser
(unchanged)
▼ Apply to VRM bones
spine · upperChest · leftUpperArm · rightUpperArm · leftLowerArm · rightLowerArm · head (tilt only)
