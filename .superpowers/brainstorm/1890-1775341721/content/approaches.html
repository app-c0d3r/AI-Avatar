3 ways to build this

Pick the architecture that feels right. All deliver the same features.

A

Extend the flat system minimal change

Keep one getPoseTargets() function, just add torso/head fields and multiply everything by intensity. Fastest to ship.

getPoseTargets(pose, intensity, t)
→ { lUAz, rUAz, lLAz, rLAz,
    spineX, headZ } × intensity

✗ all bones in one function — grows unwieldy
✗ talking gestures would need ugly if-else
✓ no structural change to GLTFAvatar
B

Layered animation system ★ recommended

Three independent layers run in useFrame and write to a shared bone-state object. Layers can run simultaneously without interfering.

LAYER 1
Pose Layer
arms + torso
+ head tilt
× intensity

procedural
gestures
during TTS
LAYER 3
Existing
blink
mouth sync
mouse look
✓ layers compose cleanly, easy to add more later
✓ talking gestures never fight emotion poses
✓ GLTFAvatar stays under ~300 lines
C

State machine + keyframes full rewrite

Avatar has explicit states (idle → reacting → talking). Pose data lives in keyframe objects. Transitions are driven by the state machine.

idle ──► reacting ──► returning
        └──► talking ──► idle

✓ most correct architecture long-term
✗ large rewrite of AvatarForms.tsx
✗ overkill for current pose count

Backend change (same for all): Chat API returns { content: "...", emotion: "happy", intensity: "medium" } as the first SSE chunk before content starts streaming.
